# PRD：自动回复质量评估流水线

## Problem Statement

业务方上线了"客服自动回复"功能两周，覆盖在线客服场景，目前没有评估机制，不确定整体质量水平。业务方要求评估回复质量，核心标准是"准确、有用、语气好、不能瞎编"，但描述模糊。评估结果将决定是否继续扩大自动回复的覆盖范围，后续需做成定期跑的自动化流程。

## Solution

构建一套自动化评估流水线，使用 LLM-as-Judge 对每条自动回复进行 7 维度评分（1-5 分），结合语义相似度作为参考指标，与人工标注做交叉验证，输出 HTML/Markdown 评估报告。支持 Mock 模式切换，方便验证流水线本身。

## User Stories

1. As a 业务方, I want to 看到每条自动回复的评分（准确、有用、语气、幻觉），so that 我能判断哪些回复好、哪些不好
2. As a 业务方, I want to 看到整体得分分布和各维度统计，so that 我能快速了解自动回复的整体质量水平
3. As a 业务方, I want to 看到最差的 3 条 case 及其分析，so that 我能优先关注需要改进的问题
4. As a 业务方, I want to 看到评估方法的局限性说明，so that 我能理解评分结果的可信度边界
5. As a 评估开发者, I want to 使用 Mock 模式快速验证流水线，so that 不需要真实 API 也能跑通流程
6. As a 评估开发者, I want to 将 LLM 评分与人工标注做交叉验证，so that 我能评估评分方法本身是否可靠
7. As a 评估开发者, I want to 用语义相似度作为参考指标，so that 我能从另一个角度验证评分合理性
8. As a 运维人员, I want to 定期运行评估流水线，so that 我能持续监控自动回复质量变化
9. As a 产品经理, I want to 基于评估报告决定是否扩大自动回复覆盖范围，so that 在提升效率的同时不降低用户体验
10. As a 客服培训人员, I want to 看到自动回复的典型问题模式，so that 我能针对性地改进回复策略

## Implementation Decisions

### 技术栈

- 语言：Go，使用标准模块结构（cmd/internal/pkg）
- 模块名：`auto-reply-evaluator`
- LLM API：OpenAI 兼容接口
- Embeddings：OpenAI Embeddings API
- 输出：HTML + Markdown 报告

### 评估维度体系（7 维度 + 1 参考）

| 维度 | 类型 | 评分方式 | 定义 |
|------|------|---------|------|
| 准确性 | 主维度 | LLM 1-5 | 回复信息是否与事实/规则一致 |
| 反幻觉 | 主维度 | LLM 1-5 | 是否编造了系统中不存在的信息 |
| 有用-主动解决 | 子维度 | LLM 1-5 | 是否主动帮用户处理，而非推给用户 |
| 有用-具体针对 | 子维度 | LLM 1-5 | 是否针对具体场景给出具体信息 |
| 有用-降低负担 | 子维度 | LLM 1-5 | 是否直接给出最简路径，而非增加操作步骤 |
| 有用-理解诉求 | 子维度 | LLM 1-5 | 是否理解用户真实痛点，而非答非所问 |
| 语气 | 主维度 | LLM 1-5 | 是否礼貌、共情、安抚情绪 |
| 语义相似度 | 参考指标 | Embedding 余弦相似度(0-1)，报告时**线性映射到 1-5**（mappedScore = 1 + similarity × 4），便于与其他维度横向对比 | 与人工参考回复的语义接近程度，不参与总分 |

### 权重分配（从客户视角）

| 指标 | 权重 |
|------|------|
| 反幻觉 | 25% |
| 有用性（4 子维度各 10%） | 40% |
| 语气 | 20% |
| 准确性 | 15% |

### LLM 调用策略

- 每条回复 1 次 LLM 调用，1 条 Prompt 输出全部 7 个维度分数
- 一致性验证：1 次 LLM 调用，将 20 条 annotator_notes 批量提取结构化判断
- 语义相似度：每条回复 1 次 Embedding 调用（20 次）

### Mock 模式

- 基于 annotator_notes 的关键词匹配映射分数：
  - 强负面（如"答非所问""推给用户"）→ 1 分
  - 弱负面（如"没有主动""让用户自己"）→ 2 分
  - 未提及 → 3 分
  - 弱正面（如"基本正确""道歉了"）→ 4 分
  - 强正面（如"主动帮用户查""直接给出明确答案"）→ 5 分
- 每个维度使用**独立的负面关键词列表**（非全局共享），防止一个维度的负面评价污染其他维度
- 反幻觉维度特殊处理：标注中未提及幻觉关键词 → 4 分（默认无幻觉）；提及 → 2 分
- 通过配置开关切换 Real/Mock 模式

### 一致性验证

- 从 annotator_notes 提取结构化判断（mentioned + direction）
- 与 LLM 评分做方向一致性比较：分数 ≤2 为 negative，≥4 为 positive，3 为 neutral
- 只对 annotator 提及的维度做一致性校验
- 统计整体一致率和各维度一致率
- **Mock 模式下一致性必须为 100%**：因为 Mock 评分和标注提取使用同一套关键词规则，逻辑上应完全一致，用于验证流水线正确性

### 报告结构

- 整体得分（加权总分 + 各维度均值）
- 各指标分布（统计图表）
- 最差 3 条 case 及详细分析
- 语义相似度验证（LLM 评分 vs 人工回复的相关性）
- 一致性验证结果（LLM 评分 vs 人工标注的方向一致性）
- 局限性分析（不一致 case 的根因分析）

### 报告展示维度合并

内部评分仍为 7 维度（4 有用性子维度），但最终报告展示时合并为 4 维度：

| 展示维度 | 计算公式 | 备注 |
|---------|---------|------|
| 准确性 | = 准确性评分 | 直接展示 |
| 反幻觉 | = 反幻觉评分 | 直接展示 |
| 有用 | = (主动解决 + 具体针对 + 降低负担 + 理解诉求) / 4 | 4 子维度算术均值 |
| 语气 | = 语气评分 | 直接展示 |

- 加权总分公式不变：反幻觉×25% + 有用×40% + 语气×20% + 准确性×15%
- 每条 case 详情中嵌套展示 4 有用性子维度明细，便于定位短板
- 一致性验证内部仍按 7 维度做，报告展示时有用性一致率 = 4 子维度一致率均值

### 模块架构

- `cmd/evaluator/main.go`：入口
- `internal/config/`：配置管理（API key、模型、权重、模式切换）
- `internal/model/`：数据结构定义
- `internal/loader/`：JSON 数据加载
- `internal/evaluator/`：LLM 评分 + Mock 模式 + Embedding 语义相似度
- `internal/validator/`：交叉验证（LLM 评分 vs 人工标注）
- `internal/report/`：HTML + Markdown 报告生成
- `internal/prompt/`：LLM Prompt 模板

### 数据流

```
JSON 数据 → Loader → Evaluator (LLM/Mock) → Scores
                                    ↓
            Embedder → Semantic Similarity → Reference Scores
                                    ↓
            Validator → Cross-Validation → Consistency Report
                                    ↓
            Reporter → HTML / Markdown Report
```

## Testing Decisions

### 测试原则

- 只测试外部行为，不测试实现细节
- 测试应能经受内部重构（改名、重组包结构不影响测试通过）

### 核心 Seam 与测试策略

| Seam | 测试内容 |
|------|---------|
| Evaluator | Mock 模式返回正确分数范围（1-5）；真实模式能正确调用 LLM 并解析响应 |
| Embedder | 验证返回向量维度正确；相似度高/低的文本对得分符合预期 |
| Validator | 方向一致性判断逻辑正确（positive/negative/neutral 映射） |
| Reporter | 输出 HTML/Markdown 文件格式正确、内容完整 |

### 测试范围

- 重点测试：Evaluator 接口（Mock 模式）、Validator 一致性逻辑、Loader 数据解析
- 典型测试：用 case_01（已知 annotator 标注"没有主动服务意识"）验证 Mock 模式输出"有用-主动解决"为低分

## Out of Scope

- 不修改自动回复系统本身
- 不接入真实客服系统获取实时数据
- 不做持续集成/定时任务配置（仅评估方法本身）
- 不做 A/B 测试或对比实验
- 不训练或微调模型

## Further Notes

- 评估方法的设计初衷是"可自动化"，因此选择了 LLM-as-Judge 方案，而非纯人工评估
- 语义相似度作为参考而非评分指标，是因为高相似度 ≠ 高质量（如 case_20 的自动回复和人工回复都提及"退货流程"，但自动回复是答非所问）
- annotator_notes 是重要的验证资源，但标注本身也有主观性，一致性验证结果需辩证看待
- 后续如有更多标注数据，可考虑用人的评分作为 ground truth 来校准 LLM 评分
