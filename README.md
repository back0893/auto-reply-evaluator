# 自动回复质量评估流水线（Auto-Reply Quality Evaluator）

## 项目概述

本项目构建了一套自动化评估流水线，用于量化评估客服场景下"自动回复"功能的质量。核心理念是 **LLM-as-Judge**：使用大语言模型（LLM）作为评判者，对每条自动回复进行多维度评分，并结合语义相似度与人工标注做交叉验证，最终输出 HTML/Markdown 格式的评估报告。

## 指标定义及理由

### 评估维度体系（7 维度 + 1 参考指标）

从业务方模糊的需求（"准确、有用、语气好、不能瞎编"）出发，将其拆解为 **7 个可量化维度**，每个维度 1-5 分：

| 维度 | 类型 | 量化方式 | 定义 |
|------|------|---------|------|
| **准确性** | 主维度 | LLM 1-5 分 | 回复信息是否与事实/规则一致 |
| **反幻觉** | 主维度 | LLM 1-5 分 | 是否编造了系统中不存在的信息 |
| **有用-主动解决** | 子维度 | LLM 1-5 分 | 是否主动帮用户处理，而非推给用户 |
| **有用-具体针对** | 子维度 | LLM 1-5 分 | 是否针对具体场景给出具体信息 |
| **有用-降低负担** | 子维度 | LLM 1-5 分 | 是否直接给出最简路径，而非增加操作步骤 |
| **有用-理解诉求** | 子维度 | LLM 1-5 分 | 是否理解用户真实痛点，而非答非所问 |
| **语气** | 主维度 | LLM 1-5 分 | 是否礼貌、共情、安抚情绪 |
| **语义相似度** | 参考指标 | Embedding 余弦相似度(0-1)，映射到 1-5 | 与人工参考回复的语义接近程度，不参与总分 |

### 维度选择理由

1. **准确性**：对应业务方"准确"要求，评估回复是否给出了正确的信息，这是客服回复的底线
2. **反幻觉**：对应业务方"不能瞎编"要求，在客服场景中编造信息会直接损害用户信任和公司声誉
3. **有用性（4 子维度）**：对应业务方"有用"要求。将"有用"拆解为 4 个子维度，是因为：
   - **主动解决**：衡量是否主动帮助而非推脱
   - **具体针对**：衡量是否针对具体问题而非泛泛而谈
   - **降低负担**：衡量是否简化用户操作路径
   - **理解诉求**：衡量是否真正理解用户意图
   - 拆解后可以精确定位回复的薄弱环节
4. **语气**：对应业务方"语气好"要求，客服语气直接影响用户满意度
5. **语义相似度**：作为参考指标，从另一个角度（向量空间距离）验证 LLM 评分的合理性

### 权重分配（从客户视角）

| 指标 | 权重 | 理由 |
|------|------|------|
| 反幻觉 | 25% | 最高权重，编造信息是客服场景的红线，零容忍 |
| 有用性（4 子维度各 10%） | 40% | 合计权重最高，体现"解决问题"是客服的核心价值 |
| 语气 | 20% | 语气影响用户体验和品牌形象，但不应高于"有用" |
| 准确性 | 15% | 信息准确是基础，但单独维度权重适中，因为"反幻觉"已覆盖部分准确性问题 |

## 评估方法

### 整体架构

```
auto_replies.json ──→ [Loader] ──→ [Scorer] ──→ [Report Generator] ──→ report.html / report.md
                         │              │
human_ref.json ──────────┤              ├──→ [Embedder] ──→ 语义相似度
                         │              │
                         └──→ [Validator] ──→ 一致性验证
```

### 1. LLM 评分（核心）

- **接口**：OpenAI 兼容的 Chat Completions API
- **策略**：每条回复 1 次 LLM 调用，1 条 Prompt 输出全部 7 个维度分数（JSON 格式）
- **Prompt 设计**：系统 Prompt 包含 7 个维度的详细 1-5 分评分标准（每档都有明确的行为描述），用户 Prompt 包含"用户问题 + 自动回复"
- **响应校验**：解析 JSON 后做 clamp 操作（1-5 范围校验），确保分数合法

### 2. Mock 模式

- 基于 `annotator_notes`（人工标注笔记）的关键词匹配映射分数
- 每个维度使用**独立的关键词列表**（强负面/弱负面/弱正面/强正面），防止一个维度的负面评价污染其他维度：

| 匹配强度 | 分数 |
|---------|------|
| 强负面（如"答非所问""推给用户"） | 1 分 |
| 弱负面（如"没有主动""让用户自己"） | 2 分 |
| 未提及 | 3 分 |
| 弱正面（如"基本正确""道歉了"） | 4 分 |
| 强正面（如"主动帮用户查""直接给出明确答案"） | 5 分 |

- 反幻觉维度特殊处理：标注中未提及幻觉关键词 → 4 分（默认无幻觉）；提及 → 2 分
- 通过 `--mock` 标志切换，方便快速验证流水线

### 3. 语义相似度验证

- 调用 OpenAI Embeddings API（`text-embedding-3-small`）获取自动回复和人工参考回复的向量
- 计算余弦相似度 (0-1)
- 线性映射到 1-5 分：`mappedScore = 1 + similarity × 4`，便于与其他维度横向对比
- 计算 Pearson 相关系数，验证 LLM 评分与语义相似度的相关性

### 4. 一致性验证（交叉验证）

- 从 `annotator_notes` 提取结构化判断（维度 + 方向：positive/negative/neutral）
- 与 LLM 评分做方向一致性比较：分数 ≤2 为 negative，≥4 为 positive，3 为 neutral
- 只对 annotator 提及的维度做一致性校验
- 统计整体一致率和各维度一致率
- **Mock 模式下一致性必须为 100%**（因为评分和提取使用同一套关键词规则）

### 5. 报告输出

- **HTML 报告**：包含整体得分、各维度均值条形图、20 条 Case 详情（可展开子维度）、最差 3 条 Case 深度分析、语义相似度验证、一致性验证结果、局限性分析
- **Markdown 报告**：纯文本版本，便于 CI/CD 集成和版本管理

## 局限性讨论

### 1. LLM-as-Judge 的主观性
- **问题**：LLM 评分受模型自身训练数据偏差、Prompt 措辞、随机性影响，同一回复不同模型可能给出不同分数
- **缓解措施**：通过语义相似度和人工标注交叉验证提供参考，但无法完全消除主观性

### 2. Mock 模式的粗糙性
- **问题**：基于关键词匹配，无法捕捉语义层面的细微差别。例如"基本正确"可能被映射为 4 分，但实际可能在 3-5 分之间的任意位置
- **适用场景**：仅适合流水线验证和快速原型，不应作为正式评估依据

### 3. 语义相似度的局限
- **问题**：高相似度不等于高质量。例如：自动回复与人工回复语义相似但包含错误信息，相似度仍会很高
- **缓解措施**：语义相似度仅作为参考指标，不参与总分计算

### 4. 人工标注的覆盖度不足
- **问题**：`annotator_notes` 并非对所有维度都有判断，部分维度一致性校验可能无法覆盖全部 case
- **影响**：一致性验证结果可能高估（只校验了标注者提及的维度）

### 5. 评分维度间的相关性
- **问题**：部分维度存在天然相关性（如"理解诉求"和"具体针对"），可能导致评分冗余
- **改进方向**：通过因子分析或主成分分析合并高度相关的维度

### 6. 单次 LLM 调用的局限
- **问题**：一次 Prompt 输出 7 个维度分数，维度之间可能存在锚定效应（先评的维度影响后评的维度）
- **改进方向**：可考虑多次调用分别评分，或使用 Chain-of-Thought 逐步推理

### 7. 权重分配的主观性
- **问题**：当前权重基于客服场景经验设定，不同业务场景可能需要不同的权重分配
- **改进方向**：支持可配置的权重，或通过 AHP（层次分析法）系统化确定权重

### 8. 评估方法的改进方法
-  **问题**：以人工标注的参考回复和分析为基准，自动回复的可能出现偏差
- **改进方向**： 优化评分 Prompt，在Prompt中增加"反例"引导，增加"是否解决用户真实问题"的追问，判断是否解决用户核心诉求

### 改进方向

1. **多模型 Ensemble**：同时使用多个 LLM 评分，取均值或中位数，降低单一模型偏差
2. **更多人工标注校准**：收集更多人工标注数据，建立评分校准曲线
3. **自定义权重配置**：支持不同业务场景下的权重自定义
4. **定期自动化运行**：集成到 CI/CD 流水线，每次自动回复逻辑更新后自动评估
